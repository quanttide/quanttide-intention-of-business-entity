# qtcloud-code：Vibe Coding 代码治理体系

## 战略意图

Vibe Coding 的本质是把代码变成黑盒。AI 降低了写代码的门槛，也放大了"长期妥协"的行业惯性——大量"看起来能跑"的垃圾代码涌入系统，技术债积累到临界点就会崩盘。

传统的解法是"逼人去理解代码"，但这在 Vibe Coding 时代不成立。**正确的解法是让 AI 去治理 AI**——架构师不再当质检员，而是制定元规则、搭建自动化闭环。

qtcloud-code 的目标是建立一套自举（self-bootstrapping）的代码治理体系，在 Vibe Coding 的生产方式下，保证代码的可维护性、可控性和可理解性。

## 功能预期

### AI 交叉审查（双向对齐机制）

**核心逻辑**：信息保真度测试。代码→文档→重写代码，两次转换后差异越大，说明源代码越混乱。

- 代码转文档：AI 读取指定模块代码，自动生成设计札记
- 文档转代码：新上下文，将上一步文档喂给另一 AI 实例，按文档重写
- Diff 校验：对比原代码与重写代码，差异超阈值时自动打上"重构"标签
- 输出：Diff 报告，量化每个模块的"混乱度"

**交付物**：`qtcloud-code cross-review <module-path>` 命令

### 独立 AI Reviewer 流水线

**核心逻辑**：每次提交代码，系统自动审查、自动出修复、自动提 Fix 分支。

- qtcloud-code 拥有独立的 AI Reviewer 流水线，与 devops 的构建部署流水线平行运行
- 审查 Prompt：检查高内聚低耦合、标出边界崩溃点、给出重构建议
- AI 重构建议作为下一个 Agent 的输入，自动提交 Fix 分支
- 形成"提交 → 审查 → 修复 → 合入"的内源性闭环

**交付物**：`qtcloud-code review` 独立流水线命令

### 边界测试自动化

**核心逻辑**：AI 写代码只覆盖 Happy Path，用系统自身能力逼出边界。

- AI 针对现有代码自动生成极端边界测试用例
- 测试挂了 → 报错信息丢回给 AI → AI 自行修复
- 引入变异测试：AI 故意改错变量（如 `>` 改为 `>=`）
    - 测试通过 → 测试用例无效 → 逼 AI 重写测试
    - 测试失败 → 测试用例有效 → 边界覆盖达标

**交付物**：`qtcloud-code mutate <module-path>` 命令

### 发布前自检闭环

**核心逻辑**：将前三者整合为独立的发布前自检流程，每次发版前系统自检自修。

- `plan` 阶段：对变更模块跑一轮交叉审查 + 边界测试
- `release` 阶段：跑一轮全量 AI 审查 + 变异测试
- 高风险模块自动输出治理报告，直到重构达标
- 体系逐步收敛，最终无需人工介入

**交付物**：`qtcloud-code check` 发布前自检命令

### 体系成熟

- 交叉审查、AI Reviewer、变异测试形成标准化闭环
- 无需人工介入代码审查，架构师只维护元规则和阈值
- 每个模块有可量化的"清晰度基线"
- 技术债不再积累，每次发布都在自动偿还

## 关键原则

1. **不引入外部 Lint 工具**，全部用 AI 自举解决，保持体系内源
2. **人不读代码**，只看 Diff 报告和趋势图表
3. **差异化阈值由系统自动校准**，根据历史数据不断收敛
4. **从局部模块开始**，逐步推广到全仓库

## 与 qtcloud-devops 的关系

qtcloud-code 与 qtcloud-devops 是**平行独立**的两个领域。

| 领域 | 职责 | 解决什么问题 |
|------|------|------------|
| qtcloud-devops | **治理** | 构建、部署、运行，让代码跑起来 |
| qtcloud-code | **创造** | 审查、测试、重构，保证代码不烂 |

两者独立运行，互不依赖。devops 管好基础设施和交付管道，code 管好代码本身的品质。共同构成 Vibe Coding 时代的工程保障。
